You are upgrading an existing SaaS web scraping platform called "ScraperCloud." The codebase is a monorepo with a React frontend (Vite) and Express.js backend (TypeScript, ESM). It uses PostgreSQL (Drizzle ORM), Redis (ioredis), and session-based auth. The frontend deploys to Vercel, the backend to Railway.

Below is the COMPLETE upgrade plan. Implement each section methodically, testing after each phase. Do NOT break existing functionality — all changes must be backward-compatible or include proper migrations.

PHASE 1: CRITICAL INFRASTRUCTURE FIXES & SECURITY HARDENING
====================================================================

1.1 — REMOVE UNUSED DEPENDENCIES
- Remove `passport` and `passport-local` from package.json (they're installed but never used). Clean up any leftover imports.

1.2 — REPLACE REDIS SCAN WITH SECONDARY INDEXES
- Current problem: `getAllJobs()` uses SCAN to iterate all `job:*` keys. This is O(N) and degrades at scale.
- Solution: Create a Redis sorted set per user: `user:{userId}:jobs` where the score is the job creation timestamp.
  - On job creation, ZADD the job ID to `user:{userId}:jobs`.
  - On job deletion, ZREM the job ID from `user:{userId}:jobs`.
  - Replace `getAllJobs(userId)` to use ZRANGEBYSCORE or ZREVRANGEBYSCORE on the user's sorted set, then pipeline HGETALL for each job ID.
  - Add a migration/backfill script that reads all existing `job:*` keys and populates the per-user sorted sets.
  - Update the in-memory Redis fallback in `server/redis.ts` to support ZADD, ZREM, ZRANGEBYSCORE, ZREVRANGEBYSCORE.

1.3 — FIX SESSION TABLE DROP RISK
- Current problem: `drizzle-kit push --force` at startup can drop the session table.
- Solution:
  - Replace the startup `push --force` with a proper migration system using `drizzle-kit generate` + `drizzle-kit migrate`.
  - Store migrations in a `/drizzle/migrations` folder committed to the repo.
  - At server startup, run `migrate()` from `drizzle-orm/postgres-js/migrator` (or equivalent) instead of push.
  - Ensure `connect-pg-simple`'s `session` table is created via an explicit SQL migration file, not auto-created.

1.4 — RATE LIMITING ENHANCEMENT
- Current: basic `express-rate-limit`.
- Upgrade:
  - Install `rate-limit-redis` and configure it to use Redis as the store (so rate limits are shared across potential future multi-process deployments).
  - Implement tiered rate limits based on subscription plan:
    - Free: 10 requests/minute to /api/job
    - Pro: 60 requests/minute
    - Business: 200 requests/minute
  - Add rate limit headers to responses (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset).
  - Add a dedicated rate limit for auth endpoints (5 attempts/15 minutes for login, 3 attempts/hour for registration).

1.5 — INPUT SANITIZATION & SECURITY
- Add `helmet` middleware to Express with a strict CSP policy.
- Add `hpp` (HTTP Parameter Pollution) protection.
- Sanitize all user inputs beyond Zod validation — strip any HTML/script tags from string fields using a lightweight sanitizer like `sanitize-html` or DOMPurify (server-side via jsdom).
- Add CSRF protection: implement a double-submit cookie pattern or use `csurf` alternative (since csurf is deprecated, use `csrf-csrf` package).
- Ensure all error responses never leak stack traces or internal details in production (create a centralized error handler middleware).
- Add request body size limit: `express.json({ limit: '10kb' })` for all routes except admin routes (100kb).

1.6 — API KEY SECURITY UPGRADE
- Hash API keys in the database using SHA-256 (store only the hash, show the full key once at creation time).
- Add API key scoping: allow users to create keys with specific permissions (read-only, create-jobs, full-access).
- Add API key expiration dates (optional, user-configurable).
- Add last_used_at timestamp tracking for each API key.
